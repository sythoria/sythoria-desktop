1. Project Overview
Sythoria is a Tauri-based React application for interacting with AI models. The architecture includes:
- React frontend with TypeScript
- Tauri IPC for backend functionality
- Context API for state management
- Custom hooks and providers

### ⚠️ **Critical Security Issues**
#### **A. Sensitive Data Exposure in localStorage (HIGH PRIORITY)**
- **Issue**: Conversations, API keys, and model configurations are stored in `localStorage`
- **Impact**: XSS vulnerabilities can expose all stored data
- **Examples**:
  - `types/index.ts`: `saveModelConfigs` saves API keys in plain text
  - `ChatContext.tsx`: Conversations stored in localStorage
  - `UIContext.tsx`: Theme preference stored in localStorage
- **Fix**:
  ```ts
  // Use Tauri's secure storage or encrypt
  import { invoke } from '@tauri-apps/api/core';
  await invoke('save_secure', { key: 'apiKey', value: encryptedApiKey });
  ```
#### **B. No Content Security Policy (CSP) (HIGH PRIORITY)**
- **Issue**: Tauri config has `"csp": null`
- **Impact**: No protection against injected scripts or XSS
- **Fix**: Add strict CSP in `tauri.conf.json`:
  ```json
  "security": {
    "csp": "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; img-src 'self' data:;"
  }
  ```
#### **C. No Data Sanitization (MEDIUM PRIORITY)**
- **Issue**: AI responses rendered directly without sanitization
- **Impact**: Potential XSS from untrusted AI sources
- **Fix**: Use DOMPurify or sanitize HTML:
  ```tsx
  import DOMPurify from 'dompurify';
  <ReactMarkdown remarkPlugins={[remarkGfm]}>
    {DOMPurify.sanitize(message.content)}
  </ReactMarkdown>
  ```
#### **D. API Keys Stored in Plain Text (MEDIUM PRIORITY)**
- **Issue**: `ModelConfig` includes `apiKey` in model configurations
- **Impact**: API keys exposed if frontend is compromised
- **Fix**:
  - Store keys only in Tauri backend
  - Pass keys via secure IPC
  - Use Tauri's `@tauri-apps/plugin-store` or `@tauri-apps/api/secure-storage`
#### **E. Hardcoded WebSocket URL (LOW PRIORITY)**
- **Issue**: `useWebSocket.ts` uses hardcoded `ws://localhost:8080/chat`
- **Impact**: Can't change endpoint without code change
- **Fix**: Make configurable via Tauri config or environment variable

### ⚠️ **Performance Issues**
#### **A. No Virtualization for Long Lists**
- **Issue**: Sidebar shows all conversations without virtualization
- **Impact**: Performance degrades with many conversations
- **Fix**: Use `react-window` or similar:
  ```tsx
  import { FixedSizeList as List } from 'react-window';
  <List height={500} itemCount={conversations.length} itemSize={50}>
  ```
#### **B. Inefficient JSON Parsing**
- **Issue**: Multiple `JSON.parse` calls on `localStorage` data
- **Impact**: Can cause GC pauses with large data
- **Fix**: Consider streaming parser or lazy parsing
#### **C. No Debouncing for Search**
- **Issue**: Sidebar search filters on every keystroke
- **Impact**: High re-render frequency with long lists
- **Fix**: Debounce search input:
  ```tsx
  const [debouncedQuery, setDebouncedQuery] = useState(searchQuery);
  useEffect(() => {
    const timer = setTimeout(() => setDebouncedQuery(searchQuery), 300);
    return () => clearTimeout(timer);
  }, [searchQuery]);
  ```
#### **D. Unnecessary Re-renders in Settings**
- **Issue**: Settings component re-renders entire model list on every change
- **Impact**: Performance overhead in model editing
- **Fix**: Optimize with `React.memo` or separate components

### ⚠️ **Maintainability Issues**
#### **A. No Error Boundaries**
- **Issue**: Error handling only in console
- **Impact**: App crashes on critical errors
- **Fix**: Add React error boundaries:
  ```tsx
  class ErrorBoundary extends React.Component<{}, { hasError: boolean }> {
    state = { hasError: false };
    static getDerivedStateFromError() { return { hasError: true }; }
    componentDidCatch(error: Error, info: ErrorInfo) { /* log to tracker */ }
    render() { return this.state.hasError ? <FallbackUI /> : this.props.children; }
  }
  ```
#### **B. Mixed State Management**
- **Issue**: Chat state split between `App.tsx` and `ChatContext.tsx`
- **Impact**: Confusion about state location and responsibility
- **Fix**: Consolidate chat state in `ChatContext` only
#### **C. No Centralized API Configuration**
- **Issue**: API URLs hardcoded in multiple places
- **Impact**: Harder to change endpoints
- **Fix**: Centralize in config or Tauri settings:
  ```ts
  export const API_CONFIG = {
    chatStream: "/api/chat-stream",
    wsEndpoint: "ws://localhost:8080/chat"
  };
  ```
#### **D. No Tests**
- **Issue**: No unit/integration tests visible
- **Impact**: Hard to refactor or maintain long-term
- **Fix**: Add Jest/Vitest setup and test components
#### **E. Inconsistent Mock Data**
- **Issue**: ChatContext provides mock models and title generation
- **Impact**: Can drift from App.tsx logic
- **Fix**: Centralize in a utility or shared config
#### **F. No Logging Framework**
- **Issue**: Uses `console.error` directly
- **Impact**: Hard to track errors in production
- **Fix**: Use Tauri logger:
  ```ts
  import { log } from '@tauri-apps/api/log';
  await log.error('Failed to save conversations', { error: e });
  ```

5. Architecture Issues
⚠️ Architecture Concerns
A. Tight Coupling Between Components and State
- Issue: Components directly import and mutate state in contexts
- Impact: Harder to test and reuse components
- Fix: Pass read-only props and use callbacks
B. No Global Store
- Issue: State split across multiple contexts (ChatContext, ConnectionContext, UIContext)
- Impact: Can lead to prop drilling or inconsistent state
- Fix: Consider Zustand or Jotai for global state:
    import { create } from 'zustand';
  const useAppStore = create<AppState>((set) => ({
    models: [], setModels: (models) => set({ models }),
    // ...
  }));
  
C. Fragile Model Config Management
- Issue: Model state updates are imperative and mutable
- Impact: Easy to introduce bugs when editing models
- Fix: Use immutable patterns and validation:
    const updateModel = (id: string, updates: Partial<ModelConfig>) =>
    setModels(prev => prev.map(m => m.id === id ? { ...m, ...updates } : m));
